# 6장 API와 REST API
API는 네트워크에서 프로그램 간의 상호작용을 위한 매개체이다. API는 클라이언트의 요청을 접수하여 서버에 해당 요청을 전달한다. 그러면 서버는 그 요청을 처리하여 결과물을 API에 전달하고, API는 결과물을 클라이언트에게 제공한다.

### REST API

Representational State Tranfer(REST) 는 이름으로 구분한 자원의 상태를 주고 받는 API 방식이다.

- REST API의 특징은 아래와 같이 있다.

  서버/클라이언트 구조

  무상태

  캐시 처리가능

  계층화

  인터페이스 일관성성

- RestAPI의 장단점


    | 장점 | 단점 |
    | --- | --- |
    | URL만 보고도 무슨 행동을 하는 API인지 명확하게 알 수 있다. | HTTP 메서드, 즉 GET, POST와 같은 방식의 개수에 제한이 있다. |
    | 상태가 없다는 특징이 있어서 클라이언트와 서버의 역할이 명확하게 분리된다. | 설계를 하기 위해 공식적으로 제공되는 표준 규약이 없다. |
    | HTTP표준을 사용하는 모든 플랫폼에서 사용할 수 있다. |  |

**REST API 사용 방법**

*규칙1. URL에는 동사를 쓰지 말고, 자원을 표시해야 한다.*

```java
"./students/1"

"./get-student?student_id=1"
```

위 코드는 학생 중 id가 1인 학생의 정보를 가져오는 URL 설계이다. 이 둘 중 Restful한 API 이름은 위에 것이다. 동사 사용 시 추후 개발에 혼란을 줄 수 있으며, 그 이유는 모든 사람 마다 개발 방식이 모두 다르기 때문에 자기 입맛에 맞는 동사를 사용하여 표기한다면 하나도 정리되지 않을 뿐더러 다른 사람이 볼 때 이해가 잘 되지 않을 것이다.

규칙2. 동사는 HTTP 메서드로

HTTP 메서드에는 `POST, GET, PUT, DELETE` 가 있다. 이는 각각 Create, Read, Update, Delete로 CRUD라고 부른다.

| 설명 | 적합한 HTTP메서드와 URL |
| --- | --- |
| id가 1인 블로그 글을 ***조회***하는 API | **`GET** /articles/1` |
| 블로그 글을 ***추가***하는 API | **`POST** /articles` |
| 블로그 글을 ***수정***하는 API` | **`PUT** /articles/1` |
| 블로그 글을 ***삭제***하는 API | **`DELETE** /articles/1` |

위처럼 동사를 정리함으로써, 내부적으로 동작하는 방식의 이름을 정해놓으면, 서로 다른 동작을 함수라도 조회,추가,수정,삭제를 하는 API의 종류를 쉽게 구분할 수 있다.
