### 6.1 사전 지식 : API와 REST API
#### API? 프로그램 간에 상호작용하기 위한 매개체
- 예시
> ![img_1.png](img_1.png)
> 1. 웹 사이트의 주소입력 "구글 사이트 보여줘!!"
> 2. API >> 서버에 전달
> 3. 서버 >> API가 준 최종 요청을 처리해 결과물을 만들고 API로 전달
> 4. API >> 최종 결과물을 브라우저에 전달 
> 5. 화면에서 확인 가능 <br>
> - **API는 클라이언트의 요청을 서버에 잘 전달하고, 서버의 결과물을 클라이언트에게 잘 돌려주는 역할을 함**

#### REST API(Representational State Transfer)? 웹의 장점을 최대한 활용하는 API
- 자원을 이름으로 구분해 자원의 상태를 주고방는 API방식
- URL 설계 방식
- 특징
  - 서버/클라이언트 구조
  - 무상태
  - 캐시 처리 가능
  - 계층화
  - 인터페이스 일관성
- 장점
  - **URL만 보고도 무슨 행동을 하는 API인지 알수 있음**
  - 무상태 >> 클라이언트와 서버의 역할이 명확하게 분리 됨
  - HTTP 표준을 사용하는 모든 플랫폼에서 사용 가능
- 단점
  - GET, POST와 같은 방식의 개수 제한
  - 설계를 하기 위한 공식적으로 제공되는 표준 규약이 없음
- 사용 방법
    - 규칙 1. URL에는 동사를 쓰지 말고 자원을 표시해야 함
        - 여기서 자원은 가져오는 데이터를 뜻함
        - 동사를 사용하면 추후 개발에 혼란을 줄 수 있음

      > 예시) 데이터를 요청하는 URL을 설계할 때  
      > 어떤 개발자는 get, 다른 개발자는 show를 사용하게 될 경우

      | 예문                  | 적합성 | 설명 |
          |-----------------------|--------|------|
      | /students/1           | 적합   | 1번 학생 자원을 조회 |
      | /ger-student?student_id=1 | 부적합 | 자원이 아닌 다른 표현을 섞어 사용 |
      | /articles/1           | 적합   | 동사 없음, 1번 글을 가져온다는 의미가 명확 |
      | /articles/show/1     | 부적합 | show라는 동사가 포함됨 |
      | /show/articles/1     | 부적합 | show라는 동사가 포함됨 |

    - 규칙 2. 동사는 HTTP 메서드로 표현
        - HTTP 메서드
        - CRUD

      | 메서드 이름 | 설명 |
          |------------|------|
      | POST       | 만들기 (Create) |
      | GET        | 읽기 (Read) |
      | PUT        | 업데이트 (Update) |
      | DELETE     | 삭제 (Delete) |
