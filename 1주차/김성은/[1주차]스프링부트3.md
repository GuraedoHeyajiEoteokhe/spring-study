## 01장. 자바 백엔드 개발자가 알아두면 좋은 지식

**클라이언트**(서버로 요청하는 프로그램) <-> **서버**(클라이언트의 요청을 받아 처리하는 주체)

**데이터베이스**  
-RDB(관계형 데이터베이스): 데이터를 행과 열로 이루어진 테이블로 관리, 기본키를 사용해 각 행 식별  
-NoSQL: 정해진 테이블 구조(스키마) 없이 자유로운 형태로 데이터를 저장하는 데이터베이스

**아이피**: 서버를 찾기 위한 번호  
**포트**: 서버에서 운용되고 있는 서비스를 구분하기 위한 번호  
-> 스프링 부트 서버 포트 기본값은 8080

**라이브러리**: 클래스, 함수 등을 모아놓은 코드의 모음 (의존성에 추가)  
**프레임워크**: 소프트웨어 개발 환경  
-> 차이점: 프레임워크는 전체적인 구조 잡기, 라이브러리는 필요한 기능 구현

백엔드 개발자의 업무: 서버 측 애플리케이션을 개발하는 일  
-과제 할당 -> 과제 분석 -> 개발 -> 테스트 -> QA 및 버그 수정 -> 배포 -> 유지보수  
-빌드에서 배포까지의 과정 자동화 가능 -> CI(지속적인 통합), CD(지속적인 배포)

자바 어노테이션: 보통 메타 데이터(데이터에 관한 데이터)로 사용하는 경우가 가장 많음  
-**@Override**: 메서드 재정의, 부모 클래스의 메서드를 자식 클래스에서 다시 구현  
-**@Deprecated**: 더 이상 사용되지 않음  
-**@SuppressWarnings**: 컴파일 경고를 무시함  
-**@Autowired**: 스프링 컨테이너에 있는 빈을 주입하는 역할  
-**@ResponseBody**: 리턴값을 View로 보내지 않고 HTTP 응답으로 직접 전송  
-**@Component**: 클래스를 빈으로 등록  
  -> @Configuration, @Repository, @Controller, @RestController, @Service 모두 @Component 어노테이션을 가지고 있음  
  -> @Component는 클래스 자체를 자동 스캔해 빈 등록, @Bean은 개발자가 메서드로 직접 생성해 빈 등록  
-**@RestController**: 라우터 역할(클라이언트의 요청에 맞는 메서드 실행), @Controller + @ResponseBody  
-**@SpringBootApplication**: 스프링 부트 사용에 필요한 기본 설정  
  -> @SpringBootConfiguration: 스프링 부트 관련 설정(@Configuration 상속)  
  -> @ComponentScan: 사용자가 등록한 빈(@Component라는 어노테이션을 가진 클래스)을 읽고 등록  
  -> @EnableAutoConfiguration: 자동 구성 활성화  
-**@NoArgsConstructor**: 매개변수가 하나도 없는 기본 생성자를 자동 생성  
-**@AllArgsConstructor**: 모든 필드를 매개변수로 받는 생성자를 자동 생성  

## 02장. 스프링 부트 3 시작하기

**스프링**: 엔터프라이즈 애플리케이션(대규모의 복잡한 데이터 관리)을 쉽게 개발할 수 있도록 도와주는 프레임워크  
**스프링부트**: 스프링 프레임워크를 더 쉽고 빠르게 이용 가능하게 만들어주는 도구(스프링에 속함)  
-> 차이점: 스프링은 톰캣과 같은 WAS(웹 어플리케이션 서버)에서 배포, 스프링부트는 WAS를 자체적으로 소지

**IoC(제어의 역전)**: 다른 객체를 직접 생성하거나 제어하는 것이 아니라 외부에서 관리하는 객체를 가져와 사용하는 것  
**DI(의존성 주입)**: 제어의 역전을 구현하기 위해 사용하는 방법, 외부에서 객체를 주입받아 사용하는 것  
**AOP(관점 지향 프로그래밍)**: 핵심 관점, 부가 관점으로 나누어서 관심 기준으로 모듈화(핵심 관점 코드에만 집중, 변경과 확장 용이)  
**PSA(이식 가능한 서비스 추상화)**: 어느 기술을 사용하던 일관된 방식으로 처리하도록 하는 것  
-> 스프링 프레임워크는 IoC/DI를 통해 객체 간의 의존 관계를 설정하고, AOP를 통해 핵심 관점과 부가 로직을 분리해 개발하며, PSA를 통해 추상화된 다양한 서비스들을 일관된 방식으로 사용하도록 함

스프링 컨테이너: 빈이 생성되고 소멸되기까지의 생명주기를 관리  
빈: 스프링 컨테이너가 생성하고 관리하는 객체

스프링 부트 스타터: 의존성이 모여 있는 그룹  
자동 구성: 자동 설정은 META-INF에 있는 spring.factories 파일에 담겨 있음

## 03장. 스프링 부트 3 구조 이해하기

스프링 부트3 구조: 각 계층은 서로 소통할 수는 있지만 다른 계층에 직접 간섭하거나 영향을 미치지 않음  
-**프레젠테이션 계층(Controller)**: HTTP 요청을 받고 이 요청을 비즈니스 게층으로 전송  
-**비즈니스 계층(Service)**: 모든 비즈니스 로직 처리  
-**퍼시스턴스 계층(Repository)**: 모든 데이터베이스 관련 로직 처리 -> DAO(데이터베이스 계층과 상호작용하기 위한 객체) 객체 사용 가능

application.yml 파일: 스프링 부트 설정, 서버 실행 시 자동 로딩  
-> show-sql, format-sql 옵션: 데이터베이스에 쿼리할 일이 있으면 실행 구문을 모두 보여주는 옵션  
-> defer-datasource-initialization: 테이블을 생성하고 data.sql 파일에 있는 쿼리를 실행하도록 하는 옵션

build.gradle: 의존성 추가 (JPA, H2, 롬복 등 추가 가능)  
-> implementation, testImplementation, runTimeOnly, compileOnly, annotationProcesser 같은 구성들이 있는데 사용 시점에 따라 다르게 사용

매핑 작업에는 인터페이스 파일이 필요 -> Repository는 대부분 인터페이스 형태

**스프링 부트 요청-응답 과정 정리**  
1.포스트맨에서 톰캣에 GET 요청, 이때 디스패치 서블릿이 URL 분석 후 요청을 처리하는 컨트롤러를 찾아서 GET 요청 전달  
2.해당 메소드와 요청 매치, 메서드에서는 비즈니스 계층과 퍼시스턴스 계층을 통하면서 필요한 데이터 가져옴  
3.뷰 리졸버는 템플릿 엔진을 사용해 HTML 문서 또는 JSON, XML 등의 데이터 생성  
4.결과를 리턴하고 그 데이터를 포스트맨에서 조회 가능
